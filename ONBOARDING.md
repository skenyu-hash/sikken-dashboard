## 1. このプロジェクトは何か
SIKKEN Dashboard（経営OS）— SIKKENグループ7社の経営数字を一元管理するWebアプリ。
- 本番: https://sikken-dashboard.vercel.app/
- 水道・電気・鍵・ロード・探偵の5業態 × 全国エリアの売上・粗利・KPIをダッシュボード化
- データ入力 → 集計 → ダッシュボード → 会議資料 を一つのアプリで回す
- 目的: 広告依存からの脱却、KPI標準化、IPO/M&Aを見据えた経営基盤づくり

非エンジニアの反専務が独学で立ち上げ、Claudeと二人三脚で開発してきたプロジェクト。

## 3. 開発体制 — どう作業が回っているか（最重要）

このプロジェクトは独特の体制で動いています。これを理解すれば、あとは流れに乗るだけです。

### 登場人物
- **あなた（と反専務）= 監督**: 「何をしたいか」を自然言語で指示し、承認する。コードは書かない。
- **Claude Code (CC) = 実装者**: VSコード内のAI。調査・計画・実装・テスト・git を全部やる。
- **3つの番人エージェント**: CCが実装した後、別人格のAIが検証する
  - `implementer`: 実装担当
  - `number-verifier`: 金額・粗利・率を独立検算する番人（実装を信用せず手計算で照合）
  - `invariant-guard`: 触ってはいけないもの(絶対不変項目)への違反を監視する番人

### 作業の流れ（標準ループ）
1. あなたが自然言語で指示（例:「ダッシュボードに○○を追加したい」）
2. CC が **Step 1（調査+計画）** を提示 → あなたが承認
3. CC が実装＋自己検証
4. CC が番人(number-verifier / invariant-guard)を招集して検証
5. 番人の指摘を解消（指摘がなくなるまで反復）
6. CC が PR を作成（本文も自動生成）
7. あなたが本文を読んで「マージOK」 → CC がマージ

**なぜこんな体制か**: 会社の経営数字(粗利・売上)を扱うので、1円の誤りも許されない。だから「実装するAI」と「検算するAI」を分けて、互いにチェックさせている。実際、過去に検算ミスを番人が見つけて防いだ実績がある。

### あなたが守るべき2つの原則
- **マージ（本番リリース）は、必ず人間(監督)が「OK」と言ってから**。CCが勝手にマージしない設定（CLAUDE.md §6 案A）。マージ=経営数字を本番に出す経営判断。
- **「絶対不変項目」(CLAUDE.md §1)には触らせない**。特に2026年4月以前のデータは、過去の数字が変わると重大事故。

## 4. 技術スタック（🔧 実装補助タイプ向け）

- **フレームワーク**: Next.js (App Router) ※バージョンが新しく breaking changes あり。CLAUDE.md 冒頭の警告を読むこと
- **ホスティング**: Vercel（mainにマージ → 自動デプロイ、30秒〜1分で反映）
- **DB**: Neon Postgres（project `red-lake-36460896`）
- **テスト**: tsx スクリプト（`scripts/test-*.ts`、`npm run test:integration:*`）。現在300件超pass
- **言語**: TypeScript

### コードを触る時の鉄則（CLAUDE.md に全部ある）
- 大規模変更は slice 分割、1コミット1テーマ
- ブランチ命名: `feature/` `fix/` `chore/`
- 金額・粗利の計算を変えたら、必ず実データで手計算照合（number-verifier に検算させる）
- 絶対不変項目に触れる時は必ず監督に確認

### 絶対不変項目（TOUCH厳禁・CLAUDE.md §1 参照）
- `AUTOSAVE_DISABLED_C89_P1`
- **2026年4月以前のデータ**（過去の数字が遡って変わるのは重大事故）
- `calculations.ts` の camelCase `vehicleCount`
- c94 / c95 系の既存機能
- 既存テストを緑のまま保つ

## 5. セットアップ手順

### 必要なもの
1. **GitHub アカウント** → リポジトリ `skenyu-hash/sikken-dashboard` への招待を反専務から受ける
2. **VSCode**（エディタ）
3. **Claude Code**（VSCode内で動かすAI。自分のAnthropicアカウントが必要。反専務のアカウントは共有しない）
4. （実装補助タイプのみ）**Node.js** 環境、**gh CLI**（`gh auth login` で自分のGitHubを認証）

### 手順
1. VSCode と Claude Code をインストール
2. リポジトリをクローン: `git clone https://github.com/skenyu-hash/sikken-dashboard.git`
3. VSCodeでフォルダを開く
4. Claude Code を起動し、§2 の「最初にやること」を投げる

### 権限について（段階的に渡される）
本番に影響する権限は、いきなり全部は渡されません。役割と信頼に応じて段階的に：
- **GitHub**: 最初から（コードを見る・PRを作る）
- **本番DB(Neon)書き換え**: 当面は反専務のみ。あなたは触らない（CCに任せ、実行は承認制）
- **本番デプロイ(Vercel)**: 同上

これは事故防止です。慣れるまでは「実装・提案はするが、本番反映は監督承認」というクッションを置きます。

## 6. 困った時

- **プロジェクトの現状が分からない** → CCに「CLAUDE.md §7 の進行状況を教えて」
- **これを触っていいか不安** → CCに「これは絶対不変項目に触れる？」と聞く。invariant-guard が判断
- **数字が合っているか不安** → CCに「number-verifier で検算して」
- **本番でトラブル** → RUNBOOK.md を見る（緊急時の復旧手順）
- **なぜこの実装になっているか分からない** → DECISIONS.md を見る（判断の経緯と却下案）
- **大きな判断・経営に関わる相談** → 反専務に確認。重大局面（CLAUDE.md の第三者レビュー条件）では別途 Web Claude で第三者視点の相談も可
- **CCが暴走しそう/おかしな方向に行った** → 「一旦止めて。Step 1 の調査からやり直して」でリセット

### やってはいけないこと
- CCに「全部おまかせで一気にやって」と丸投げする（暴走の元。必ず Step 1 調査 → 承認 → 実装 の順を踏ませる）
- マージ（本番リリース）を監督承認なしで実行する
- 絶対不変項目を「たぶん大丈夫」で触る
- **⚠️ water(水道)の2026年5月以降データを /import-monthly(Excel)経由で投入する** → コンサル費控除が抜け（import-monthly が aggregation を経由しないため）粗利が7.7%過大表示される（c95-B-5で対応するまで禁止。詳細は KNOWN_ISSUES.md §8）
