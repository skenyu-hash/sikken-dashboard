## 0. このドキュメントの目的

水道フォーム再設計を皮切りに、5業態（水道・電気・鍵・ロード・探偵）の日次入力フォームを再構築する。本文書は実装の単一情報源 (SSOT) として機能する。

仕様変更が発生した場合は本文書を更新し、関連 PR から本文書を参照すること。

---

## 1. 背景と方針

### 1.1 解決する課題

1. **JSON ペースト方式の入力負担**：従来の `/import` は写真→AI読取→JSON という間接ルートで、日次運用に向かない
2. **権限制御の二重管理**：UI 側 `CAN_EDIT_DASHBOARD` 配列と API 側 `canEditArea` でロール定義が二重化
3. **フィールドの重複・曖昧性**：「追加」「ヘルプ」「HELP」が混在、件数の集計関係が不明瞭
4. **業態間のミスマッチ**：水道用フィールドが鍵・ロード・探偵にも表示される

### 1.2 設計方針

- **日次フォーム主体運用**：日々の入力はダッシュボード下部のフォームから行う
- **`/import` は維持**：AI写真読取・一括投入用途で残す
- **5業態それぞれに独立フォーム**：構造は同一だが業態ごとに語尾を調整可能
- **権限の SSOT 化**：A-1（データスコープ）と A-2（ページアクセス）を `lib/permissions.ts` に集約
- **段階移行**：feature flag で新旧フォーム切替、移行期間中は両方動作

## 5. 業態展開方針

### 5.1 全業態共通

31フィールド構成は5業態すべて共通。フィールドの個数・順序・自動計算式は同一。

### 5.2 業態別の差分（フィールド名語尾のみ）

| 業態 | DB値 | 「工事」表記 | 備考 |
|---|---|---|---|
| 水道 | water | 工事 | 標準形 |
| 電気 | electric | 工事 | 水道と同一 |
| 鍵 | locksmith | 工事 | 水道と同一 |
| ロード | road | 出動 | "工事"→"出動" に置換 |
| 探偵 | detective | 調査 | "工事"→"調査" に置換 |

### 5.3 用語マッピング表（実装時参照）

水道版を canonical として、他業態は以下のマッピングを適用：

| 水道版ラベル | 電気・鍵 | ロード | 探偵 |
|---|---|---|---|
| 総工事件数 | 同左 | 総出動件数 | 総調査件数 |
| 外注工事件数 | 同左 | 外注出動件数 | 外注調査件数 |
| 自社工事件数 | 同左 | 自社出動件数 | 自社調査件数 |
| 外注工事費 | 同左 | 外注出動費 | 外注調査費 |
| 自社工事利益 | 同左 | 自社出動利益 | 自社調査利益 |
| 実質工事コスト | 同左 | 実質出動コスト | 実質調査コスト |

その他のフィールド名（売上・対応件数・コスト・広告費・HELP・粗利）は全業態同一。

### 5.4 実装上の方針

**第一段階**：水道版フォームを5業態に同一展開（全エリア・全業態で水道の用語をそのまま表示）。これは即座に運用開始するため。

**第二段階**：業態別の用語マッピングを `lib/business-terms.ts` に集約し、フォームコンポーネントが業態を引数に受け取って用語を切り替える方式に拡張。

**第三段階**：各業態責任者からのフィードバックで、必要に応じて業態固有のフィールドを追加・削除。

```

            ├── PR #37 (new form water)
```

### 6.2 PR順序の決定理由

- **PR #35 を最優先**：権限の SSOT がないと、他PRが二重管理を悪化させる

- **PR #38 で feature flag**：本番運用を維持しながら新フォームを並行稼働
- **PR #41 で 5業態展開**：水道版の安定後、横展開
- **PR #43 で旧フォーム削除**：全 5業態運用が安定してから

## 7. ガードレール（守るべき制約）

### 7.1 既存データを壊さない
- entries テーブルの既存レコードは保持
- 新規列は NULLABLE で追加し、後方互換性を維持
- 旧フォームから投入された過去データはそのまま参照可能

### 7.2 既存ユーザー46名のログインを壊さない
- 認証フローは変更しない
- 権限定義の SSOT 化はリファクタのみで挙動変更なし

### 7.3 `/import` は維持
- AI写真読取・一括投入用途で残す
- 新フォームと同じ entries テーブルに書き込む（衝突時は後勝ち）

### 7.4 段階的移行
- feature flag (`NEXT_PUBLIC_NEW_FORM_ENABLED`) で新旧切替
- 移行期間中は新旧両方動作
- 全業態運用安定後に旧フォーム削除

### 7.5 監査ログ
- 入力・編集の操作ログを記録（誰が・いつ・何を変更したか）
- A-1 例外（部長以上の他エリア閲覧）も記録対象
